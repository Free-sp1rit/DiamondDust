# Execution Plan: Local Trial Run Log Scope

## Product Goal

Make provider-free local trial AI run logs explicit about fixture scope, no-provider execution, metrics non-applicability, and downstream trial lineage without implying real LLM quality validation, user acceptance, formal vault apply, or publication readiness.

## Current Understanding

The local trial run log already records required AI run fields such as `run_id`, task/provider/model metadata, hashes, validation status, and `created_at`. Product-owner feedback correctly identifies that the persisted artifact can be misread as a real provider run unless it states that the run is fixture-driven and provider-free.

This task uses the product owner's two temporary trial-feedback principles only for this feedback task. They are not long-term governance unless separately approved.

## Assumptions

- The generic AI run log renderer should remain provider-neutral and should not require local trial fields for all run logs.
- Local trial-specific fields can be added through a typed storage artifact context instead of untyped domain JSON.
- `prompt_version` can remain `extract_units.v1` as the task contract reference while `prompt_used: false` clarifies that no real prompt execution occurred.
- `cost` and `latency` should remain `null`; a `metrics_scope` object should explain why those metrics are not applicable.

## Non-goals

- Do not call a provider.
- Do not invent real cost or latency.
- Do not create new trial artifacts or new artifact paths.
- Do not change formal vault files.
- Do not record patch acceptance, formal write approval, user acceptance, or publication approval.
- Do not make the temporary feedback principles permanent governance.

## Feedback Evaluation

- Add local trial identity and provider-free fixture scope fields: accepted. These improve artifact semantics and reduce false interpretation risk.
- Add `prompt_used: false` while preserving `prompt_version`: accepted. This keeps the contract reference while making no-prompt execution explicit.
- Add `metrics_scope`: accepted. This is clearer than inventing zero cost or latency.
- Add lineage fields: accepted with structure. `source_input_id` should be top-level, and `output_artifacts` should point to the final local trial report/outcome artifacts generated by the same trial.
- Add `not_validated`: accepted. This aligns with the provider-free quality boundary and makes machine-readable scope safer.

No feedback is rejected at plan time. The implementation should preserve semantically distinct fields such as extraction `validation_status`.

## Proposed Technical Approach

Add typed storage-layer run log context dataclasses and allow `render_ai_run_log_artifact` / `write_ai_run_log_artifact` to merge optional context into persisted JSON. Update local trial orchestration to pass fixture scope context when writing the run log. Update tests, artifact contract docs, repo memory, and milestone review. Regenerate the existing ignored local trial artifacts for inspection without creating new artifact paths.

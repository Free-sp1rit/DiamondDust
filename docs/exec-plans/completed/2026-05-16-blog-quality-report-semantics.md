# Execution Plan: Blog Quality Report Semantics

## Product Goal

Make provider-free local trial blog quality reports precise enough for product-owner review without implying publication readiness, real AI generation quality, formal vault writes, or user acceptance.

## Current Understanding

The current blog quality report is a body-only Markdown artifact with an ambiguous `status` summary and `none` risk/unsupported-claim lines. Related artifacts already separate local trial pipeline success from product-owner verdict and fixture-driven scope. The blog quality report should align with that boundary while staying an AI report artifact under `_ai_reports/blog-quality/`.

## Assumptions

- The accepted feedback is beneficial because it clarifies artifact semantics and review boundaries.
- Generic blog quality reports can safely gain frontmatter and `quality_status` wording without adding publication scoring.
- Local trial fixture scope should be represented through contextual report metadata rather than hardcoded global behavior.

## Non-goals

- Do not call a provider.
- Do not create a publication decision artifact.
- Do not introduce scoring or publication readiness scores.
- Do not modify formal vault files.
- Do not record user acceptance.

## Proposed Technical Approach

Add blog quality report context metadata to the storage renderer, use it from the local trial harness, and keep generic report output provider-neutral. Update tests, artifact contract docs, and the existing local trial blog quality report so Markdown and generator semantics agree.
